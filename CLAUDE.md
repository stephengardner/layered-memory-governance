## Decisions

- The LAG Console backend maintains an in-memory atom index (Map<filename, Atom>) primed on startup and kept coherent via the filesystem watcher. This index is a PROJECTION per canon `arch-atomstore-source-of-truth` — the .lag/atoms/*.json files remain the sole source of truth. The index is a performance optimization: it takes read requests from O(N disk reads) to O(1) memory lookups, which matters at the org-ceiling of 50 concurrent actors per `dev-indie-floor-org-ceiling`. Any future server (SQLite, Postgres, Redis) that replaces the in-memory map MUST preserve this invariant: disk is canon, the index mirrors disk, disk wins on any coherence conflict. A crash + restart replays from disk via primeAtomIndex(). Writes go to disk first; the watcher updates the index on the filesystem event, not on the write path. _(confidence 1.00)_
- Atoms proposed through any UI surface (Console, future Slack command, future Tauri app) are written at layer L0 with `validation_status: pending_review`. L3 promotion requires the existing human-approval flow per `inv-l3-requires-human` + `pref-l3-threshold-default`. The proposal endpoint (POST /api/atoms.propose) is INTAKE, not auto-canonization — it opens a new door into the governance pipeline, it does not bypass the gate. A propose-to-L3 path would compromise the three-layer architecture; a propose-to-L0-pending path integrates cleanly with existing promotion logic. _(confidence 1.00)_
- LAG's HTTP API surface (POST /api/canon.list, /api/canon.applicable, /api/atoms.references, etc.) is the canonical READ surface for every consumer — agents AND the Console UI. Agents MUST NOT reach into .lag/atoms/ or .lag/principals/ directly to query governance state; they call the API. This makes LAG an actual substrate with a stable contract rather than a filesystem convention. The API envelope ({ ok: true, data: T } / { ok: false, error: { code, message } }) is the versioned interface; breaking changes must go through a new endpoint, not a silent field rename. The POST /api/canon.applicable endpoint in particular is the agent-facing governance query — given {principal_id, layer, scope}, it returns the canon atoms that govern that position, ranked by the source-rank formula. Agents who duplicate arbitration logic are drift waiting to happen. _(confidence 1.00)_
- Card-grid layouts in the LAG Console that need a masonry look (cards of varying heights pack tightly) use a two-stack CSS-grid pattern: the grid has two 1fr columns, and render-order items are distributed into left/right flex stacks by index parity (even → left, odd → right). Each stack is an independent flex column so expanding or collapsing a card grows only its own column; no sibling re-layout, no visible jump. This is the DECISION for every masonry-shaped grid in this codebase unless a future requirement forces otherwise (explicit supersession). Plans view is the first consumer; future activity-feed variants or atom-detail grids inherit the same pattern. _(confidence 1.00)_
- CodeRabbit (and similar SaaS reviewers) hardcode a bot-comment anti-loop: @-mentions from [bot] accounts are silently ignored. For canon directive dev-coderabbit-required-status-check-non-negotiable to hold without forcing the operator's personal account into every automation flow, we route CR-trigger comments through a purpose-built machine-user principal (GitHub User-type account, peer to lag-ceo/lag-cto/lag-pr-landing bot identities). Scope: machine user ONLY for CR trigger comments; all other automation (PR open, merge, review replies, nit resolution) continues through GitHub App bot identities. Concrete form: ReviewTriggerAdapter interface + UserAccountCommentTrigger impl + LAG_OPS_PAT secret wired into scripts/run-pr-landing.mjs as a pre-check. _(confidence 1.00)_
- lag-ceo[bot] is the operator's human-proxy GitHub identity and is attribution-only. It opens PRs, posts operator-initiated comments, and merges when the operator delegates. It MUST NOT carry decision-bearing authority. Decisions that change canon, policy, or architectural shape are lag-cto's responsibility; lag-ceo is the GitHub-side attribution layer for the operator's proxy actions. The distinction matters because collapsing the two would make git history lie about where a decision came from: a PR opened by lag-ceo is 'operator said do this', a PR opened by lag-cto is 'CTO planning loop produced this plan'. The roles have the same permission set on GitHub (same API surface needed) but distinct semantics in the provenance chain and authority hierarchy. Operator stated the role distinction explicitly in session 2026-04-20 when designing the three-bot identity model (Model B + lag-ceo addition). _(confidence 1.00)_
- Inbox V1 is the inter-actor messaging primitive: actor-message atoms in the AtomStore plus Scheduler-driven pickup plus policy-atom ordering. Not a new Host sub-interface. _(confidence 1.00)_
- Actors acquire per-role GitHub App identities via the lag-actors CLI. Terminal automation routes through gh-as.mjs using short-lived installation tokens. The operator personal gh auth is not used for autonomous actions. _(confidence 1.00)_
- The code-author blast-radius fence (ADR design/adr-code-author-blast-radius-fence.md) reserves the pol-code-author-* policy-atom slot. Graduation criteria are documented; fence atoms are seeded only when the code-author principal materializes. _(confidence 1.00)_
- plan_state is a top-level Atom field, not a metadata key. Surfaced by the first self-audit run when PlanningActor was writing it into metadata where dispatch and auto-approve loops could not see it. Consumers read atom.plan_state, never atom.metadata.plan_state. _(confidence 1.00)_
- Direct session-agent queries to GitHub for PR state are prohibited. The pr-landing actor is the canonical PR observer; its output (escalation actor-message atom + PR comment delivery) is the source of truth a session agent consumes. When fresh state is needed, the agent invokes the actor via scripts/run-pr-landing.mjs (observe-only mode is a follow-up); the invocation itself is audited through the operator-action atom trail. Short-term bridge: scripts/pr-status.mjs runs the same composite read the actor uses, without the full actor loop. Enforcement layer: .claude/hooks/enforce-pr-status-composite.mjs blocks ad-hoc `gh pr view` / `gh api .../pulls/<N>` state queries at the PreToolUse level and redirects to pr-status.mjs. Motivation: two in-session failures (2026-04-20 and 2026-04-21) where the agent polled one surface, missed CodeRabbit review completion on another, and reported stale state; the directive layer (dev-multi-surface-review-observation) proved insufficient without a mechanism. _(confidence 1.00)_
- The autonomous-merge endgame is policy-driven auto-merge by a bot, NOT a second human co-maintainer. Model B (the lag-cto bot has merge capability, constrained to flow through plan-dispatch) is the chosen path for clean GitHub audit-log provenance. CTO direct merge under medium-tier kill switch protection is the concrete implementation. _(confidence 1.00)_
- The AtomStore is the single source of truth. Every other artifact (canon markdown, dashboards, audit reports, agent context) is a projection over the atom set. Never derive from or write to a projection directly. _(confidence 1.00)_
- Canon renders into a bracketed section of a target CLAUDE.md file via CanonMdManager. Content outside the markers is preserved byte-for-byte. Multi-target canon (one file per scope or role) composes via LoopRunner.canonTargets. _(confidence 1.00)_
- The Host interface is the sole boundary between framework logic and any concrete implementation. Eight sub-interfaces: AtomStore, CanonStore, LLM, Notifier, Scheduler, Auditor, PrincipalStore, Clock. LAG logic never reaches around this boundary. _(confidence 1.00)_
- The Notifier is a pluggable channel seam, not a hardcoded file queue. File queue is the V0 implementation; Telegram, Slack, session-inject, and email are channel implementations behind the same interface. Callers never branch on channel type. _(confidence 1.00)_
- Principals form a hierarchy via signed_by. Authority cascades from root down. Arbitration respects the chain (source-rank uses principal depth as a tiebreaker). Taint propagates transitively when a leaf is compromised. _(confidence 1.00)_

## Preferences

- Source-rank scoring formula: Layer x 10000 + Provenance x 100 + (MAX_PRINCIPAL_DEPTH - depth) x 10 + floor(confidence x 10). Layer dominates provenance dominates hierarchy dominates confidence; confidence only breaks ties within a layer. _(confidence 1.00)_
